JOB TICKET SAMPLE

 

Summary

This sample demonstrates how to write a Unidrv-based ui plug-in that implements the native Job Ticket and DevCap Provider Interface to support the new Job Ticket architecture provided in Longhorn release of Windows Operating System. This version of the sample produces ui plug-in dll that implements the Job Ticket Provider Interface. This version of the sample will not build with the Windows 2000, Windows XP or Windows Server 2003 build environment since it uses msxml6 parser which is not present on pre-Longhorn Operating Systems.

OEMJTProvider Module

The OEMJTProvider module in the ui plug-in is the basic module that implements the methods of IPrintOemJobTicketProvider Interface to customize the behavior of core driver’s (Unidrv) provider interface implementation. The sample plug-in demonstrates implementation of provider interface through implementation of Stapling feature. The various methods in this module demonstrate how the plug-in can populate job ticket with its own settings, interpret the setting in input Job Ticket and correspondingly make changes in its own settings, populate device capabilities document etc.

 

Job Ticket Handler Library

The jthandler module in the ui plug-in is a helper library that contains all the required XML functionality to handle processing of Job Ticket/Device capabilities. The helper library code is developed using MSXML6 XML DOM implementation and it makes use of the Plug-in Helper Interface provided by core driver (Unidrv) namely IPrintJobTicketPluginHelper Interface to create instances of XML Objects ( eg MSXML Namespace Manager etc.)

 

Implementation of Stapling Feature

The Sample plug-in demonstrates the processing of Job Ticket/Device capabilities through implementation of Stapling Feature. Stapling feature is especially interesting because it is one of the GPD features that Unidrv does not recognize and hence it provides generic handling for this feature. This means that while creating a Job Ticket from Devmode, Unidrv will not put the features like Stapling under public printschema namespace. Instead unidrv will put stapling under private namespace. Plugin gets a chance to specify the namespace, but if plugin chooses not to do so, unidrv will create the namespace based on the driver name. The disadvantage of putting stapling in private namespace is that higher level applications may not be able to interpret/understand it.However since Stapling is also a feature supported by Printschema, the sample plugin reads and interprets the Staple Feature, and transfers it to public printschema namespace so that any higher level application will be able to read and interpret the setting made for Stapling. Note that since the stapling feature is present in the gpd, it is assumed that the plugin understands its specifications and therefore the plugin can convert its private specification into a specification that matches the public schema. For example, based on the information present in GPD, when Unidrv creates the Job Ticket, the Stapling Feature is specified as < printschema_namespace_prefix:Feature name="private_namespace_prefix:Stapling" >< /printschema_namespace_prefix:Feature > but public schema specifies Stapling as < printschema_namespace_prefix:Feature name="printschema_namespace_prefix:JobStaple" >< /printschema_namespace_prefix:Feature >. So during the ConvertDevmodeToJobTicket call, the plugin moves some of settings from plugin private namespace to public namespace. While performing conversion from Job Ticket to Devmode, the plug-in needs to move the stapling settings from the public namespace to the driver private namespace. It puts the Staple Feature in the input Job Ticket back into the format that Unidrv had put it in. This is necessary because, Unidrv is ultimately going to use the setting in the private namespace to initialize the option array in unidrv’s private portion of the devmode. This option array is the one that is used by unidrv to store which of the gpd options have been chosen by the user. If Staple feature in the Job Ticket is not in a format that is expected by Unidrv, then while performing conversion from Job Ticket to Devmdoe, Unidrv will not be able to recognize it and therefore will be unable to transfer the stapling setting to Unidrv private Devmode.

 

GPD

The GPD for this sample (jt.gpd) is based on oem.gpd provided with DDK alongwith some additions. Since the sample demonstrates the implementation Provider Interface by plug-in through implementation of Staple Feature,the GPD has Staple Feature and all the features needed to support stapling functionality ( ie OutputBin, Accessory Output Bin etc) added to it. The sample plug-in is implemented to work successfully and print stapled documents on HP LaserJet 8100 Series PCL and HP Mopier 320 PCL.

 

Building the sample

To build the samples, run “build –cZ” from the JTSample directory. The resulting DLL (JTUI.DLL) will be placed in the appropriate platform directory (i.e. i386 or Alpha).

 

Installing the sample

After building the sample, you can install it using the Add Printer Wizard and supplying the JTDLL.INF that is part of the sample when you are prompted for an INF in the wizard. The Unidrv binaries required for this driver need not be copied to the local directory containing jtdll.inf.
