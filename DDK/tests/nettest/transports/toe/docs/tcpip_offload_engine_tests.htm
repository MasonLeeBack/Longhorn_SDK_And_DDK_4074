TCP/IP Offload Engine Tests for Microsoft Windows Codename “Longhorn”

Introduction

This test verifies that the TCP/IP offload engine (TOE) on a network interface card (NIC) or host-based adapter (HBA) is compliant with industry standards for TCP/IP offload devices.

Global Information

Background Information

Windows Longhorn

Device\Driver Category

Friendly Name

TCP/IP Offload Engine Tests

Description

This test verifies that the TCP/IP Offload Engine (TOE) on a network interface card (NIC) or host-based adapter (HBA) is compliant with industry standards for TCP/IP offload devices.

Installation

Installation Steps

Tests are installed with the Microsoft Windows codename “Longhorn” Driver Kit (LDK). WTT jobs install the required binaries.

File Location

NETTEST\Transports\Sparta\bin installs to c:\sparta\bin

NETTEST\Transports\Sparta\sys installs to c:\sparta\sys

NETTEST\Transports\Sparta\Scripts installs to c:\sparta_test

Removal Steps

The files are removed by the WTT jobs and the LDK uninstall application.

Environment Requirements

Required Hardware

Two computers are required. The first computer must have at least two network cards, one of which must be a TOE NIC. The second computer must have at least one network card, which must be an Ethernet card. 

Required Software

All the testing software is installed by the WTT Job. The operating system must detect and install drivers for the NIC.

Tester Knowledge Requirements

Basic networking, TCP/IP protocol, and understanding of Chimney Offload

Content Quality

Unit Test

There are no unit tests. The WTT Job must be run and results must be interpreted. If there is a problem with the test architecture, it will appear in the test results.

Diagnosability

Setup Verification

Tests do not run if setup fails.

Debugging Activation and Output

There is no option to activate additional debugging. All debugging output goes to the kernel debugger. Each test creates a WTT log and step-by-step test progress is written into the log.

Test Log Diagnosis

At the end of each test log there is a summary of results which indicates which tests have passed. If there are any failures, blocks, or warnings, search for  the keywords “FAIL”, “BLOCK” or “WARN” in the log to get the failure description. Each failure, block, or warning provides an assertion number. You can check test assertions documentation for the description of the test scenario. You can also examine the output in the log above the failure line for the details of the actions performed during the test scenario.

Debug Instructions

Each test process produces a debug output. If the kernel debugger is attached to the computers, the debug output can be examined to find more information about the failure. Also, each test creates a log of what happened during testing. There is no additional tip to debug the tests. The test scenarios are networking scenarios. Knowledge of kernel debugging is required.

Test Duration

Three hours

Reboot

Yes, the computer will reboot several times.

Default Configuration

There are no command line options for the test. It runs as default.

The controller system should have an Ethernet card connected to a private network. It is better if the NIC is assigned a static IP address as shown in the figure below.

The test system needs to have two network cards, one regular Ethernet card and one TOE card. The IP addresses should be in the same subnet as the controller system.

Interference

It is strongly recommended that both computers are connected to a private and quiet network. This dramatically improves the diagnosability of the problems.

Troubleshooting

The most common issue reported for this tool is the connectivity failures between the two computers. In the event of a connectivity failure, try the following steps:

1)        Check your user credentials on the remote computer. Both computers must be either on the same workgroup or on the same domain.

2)        Try to ping the remote computer. If you cannot ping it, check whether Internet Connection Firewall (ICF) is turned on. Turn off ICF on the remote computer to enable ping. For the tests to run, you don’t have to turn off ICF. This step is just to verify you can reach the remote computer.

1)        If you succeeded in the second step, try to execute the “net view” command targeted to the remote computer. If it fails, check your user credentials on the remote computer, net stop policyagent, and net stop sharedaccess services.

2)        If net view command succeeds and the Sparta configuration utility still fails to log in, complete the following steps to check group policy settings on the remote computer:

1.       Run Microsoft Management Console (MMC)

2.       In the Console1 dialog box, press Ctrl + M to open the Add/Remove Snap-in dialog box.

3.       Click Add to open the Add Standalone Snap-in dialog box.

4.       Select Group Policy Object Editor, click Add, and then click Finish.

5.       Click Close to close Add Standalone Snap-in dialog box, and then click OK to close the Add/Remove Snap-in dialog box.

6.       Expand the Console Root tree to Console Root\Local Computer Policy\Computer Configuration\Windows Settings\Local Policies\Security Options.

7.       Find the Network access: Sharing and security model for local accounts policy on the right.

8.       Modify the policy from Guest Only to Classic.
