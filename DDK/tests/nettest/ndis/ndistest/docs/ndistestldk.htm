NDISTest Single Adapter Test

Introduction

The NDISTest is a test suite that the Microsoft® Windows® team provides to verify network, modem, NDIS intermediate driver (IM), and Winsock Direct drivers and devices. The Windows team uses NDISTest to verify devices that are to be shipped with the operating system. Windows Hardware Quality Lab (WHQL) also uses NDISTest as part of its suite of tests for Logo and driver signing.

Description

NDISTest is a test suite that the Microsoft Windows team provides to verify network, modem, NDIS IM, and Winsock Direct drivers and devices. The Windows team uses NDISTest to verify devices that are to be shipped with the operating system. WHQL also uses NDISTest as part of its suite of tests for logo and driver signing.

Theory of Operation

The 1c_KernelCalls default script determines the kernel calls that the driver is making and then checks to see if the calls are legal. For some illegal calls, the log file also contains information on which legal functions can be substituted for illegal function calls.

Installation

Installation Steps

The test installs as part of the Longhorn Driver Kit (LDK). You can launch the test by using the LDK Test Harness job titled NDISTest Single Adapter Test.

File Location

On the test client, the test is copied to the %SYSTEMDRIVE%\NDISTest directory. When executed the first time, it creates the HKCU\Software\Microsoft\NDTest registry key.

Removal Steps

NDISTest removes itself upon completion. You can manually remove it by deleting the directory and deleting the registry key HKCU\Software\Microsoft\NDTest.

 

Environment Requirements

Required Hardware

At least one LAN/WLAN network card is required on the test client. This version of NDISTest runs only on x86 systems.

Required Software

This test requires no additional software.

Unsupported Hardware

This version of NDISTest does not work on AMD64 and IA64 machines.

Tester Knowledge Requirements

To interpret the results, you must have significant knowledge of network driver development. Very little to no networking knowledge is required just to run the test.

Content Quality

Unit Test

NDISTest is verified by using internal verification scripts.

Diagnosability

Debugging Activation and Output

NDISTest writes debug information into the results log file and also on the debugger. You can increase the level of debug information on the test machine. On the Options menu, point to Debug Output, and click Loud.

Test Log Diagnosis

At the beginning of each test log is a results summary that lists whether all tests passed. If any failures or warnings appear, use the hyperlinks to go to the actual failure. You can examine the log output above the failure line for the details of the actions that were performed during the test scenario and also possible causes of the failure.

Command-Line Arguments

The following is the set of command line options for NDISTest. HCT Test Manager uses these options, but the options are not visible or accessible when NDISTest is running from the HCT Test Manager shell. To access these options, you must launch NDISTest from the command line or from a link with the supplied options. The following is an example:

            C:\hct\testbin\i386\ndtest /u /tc:{00-02-04-06-01-02} /msg:{139.33.45.33}

 Here are the options and their meaning:

            /u                     Selects MANUAL TESTS option
            /hct                   Selects WHQL TESTS option
            /client               Selects CLIENT option
            /server              Selects SERVER option
            /tc:{GUID}         /tc:{xx-xx-xx-xx-xx-xx}

You can use either of these formats to select the TEST: device. A globally unique identifier (GUID) or medium access control (MAC) address can be supplied. Note that the leading 0 must be supplied; 0-10-4b-ca-b3-29 will not work. The options are not case sensitive. Here is an example of each:

            /tc:{C9F2D762-153C-43B3-8245-E288810B0AC8}
            /tc:00-10-4b-ca-b3-29 

You can use either of these formats to select the MESSAGE: device. Replace the GUID with an Internet Protocol (IP) address. The MAC address is the MAC address of the device that this IP passes through. The format of the mac address is the same as the /tc: option described earlier. Here is an example of each:

            /msg:{IP}
            /msg:{xx-xx-xx-xx-xx-xx}

You can select multiple support devices by providing either the GUID or MAC address. The format of the MAC address was described earlier in the /tc: option. The GUIDs can be for either local or remote devices. When the backchannel discovers the remote device and adds it to the user interface (UI), the remote device is selected. You cannot use the MAC address to select remote devices.

            /msg:{157.59.140.190}
            /msg:00-10-4b-ca-b3-29
            /support:{GUID1}{GUID2}{xx-xx-xx-xx-xx-xx}{yy-yy-yy-yy-yy-yy}

You can select multiple scripts by providing their names. If any scripts are selected, then all of the other scripts are unselected. The scripts are guaranteed to have unique names.

            /script:{scriptname.wsf}{scriptname2.wsf}

This option causes all nonerror pop-up messages to time out after 60 seconds. Errors are logged in the log file, but the tests do not stop. This option is intended for internal testing:

            /tp

These options control the driver verifier settings:

            /dvi       driver verifier ignore
            /dve      driver verifier enable
            /dvd     driver verifier disable

This option causes the Start button to be depressed automatically. The tests run, and then the NDISTest exits. Note that in server mode the server script does not end until the user stops it.

            /auto

This option causes the UI not to register the COM direct-link libraries (DLLs), not to check their versions numbers, and not to end ndinfo, ndtsession, and ndtcommws if they are running when it starts. You can use this to mix and match different versions of DLLs. It is usually used during development.

            /noreg

This option notifies NDISTest that the test devices are ISDN or Cell modems. Currently, NDISTest cannot determine this programmatically.

            /isdnmodem
            /cellmodem

This option notifies NDISTest that this is a system test, not a device test.

            /WHQLSystem
