Scope – Longhorn Test or LDK Sample

This test is serving as an LDK driver compliance test.

Global Information

Background Information:

Primary Operating System

This test is designed for Microsoft® Windows® XP and later operating systems. There is a possibility that this test will be updated with Windows codename “Longhorn”-specific test cases.

Supported Operating Systems

Windows codename “Longhorn,” Windows Server™ 2003, and Windows XP

Device\Driver Category

This test sample applies to WIA (Windows Image Architecture) drivers.

Related DDK Topics

For more information, see Windows Image Acquisition Drivers in the Still Image Devices Design Guide and topics in the Still Image Devices Reference section of the Windows DDK.

Friendly Name

The friendly name for the test sample is WIA Minidriver Test.

Description

WiaTester is a tool for the WIA Compliance test that checks all required properties, methods, and commands (defined in the WIA DDK) of a WIA minidriver. The test result is summarized and rated in the test log. All WIA minidrivers should pass this test without any failure.

Theory of Operation

WiaTester consists of three executable components: WiaTestGUI.exe, WiaTester.exe, and WiaTestLog.exe. The user typically launches WiaTestGUI.exe, which displays a user interface that calls the test engine WiaTester.exe (command line application). WiaTestLog.exe is a user interface component that displays log output from WiaTester.exe.

Installation:

Installation Steps

To install the files for the test, copy the binary to the local machine and run:

RegisterAssembly.cmd WiaPropDB.dll

The test requires WTTLOG.dll.

File Location

The file is located in the Install folder.

Removal Steps

To uninstall the files for this test, run:

RegisterAssembly.cmd /unregister WiaPropDB.dll

Then, delete the installation directory.

Environment Requirements:

Required Hardware

Standard x86-based computer, AMD 64-bit computer, or an Intel Itanium computer.

WIA device (scanner) installed.

Required Software

Standard operating system installed.

Unsupported Hardware

Non-WIA imaging devices (devices with only TWAIN data source, for example) are not supported by this test.

Tester Knowledge Requirements

A tester should have the basic understanding of WIA architecture and how WIA drivers are written and work to run this test.

Content Quality:

Unit Test

To validate the quality of the content, attach any inbox scanner (for example, HP ScanJet 6200C) and run the test; the test should pass.

Diagnosability:

Setup Verification

Does the tool provide verification that setup was completed correctly?

The tool does not provide verification that setup was completed correctly. The LDK job is structured in such way that the test will abort if a setup step fails.

Debugging Activation and Output

How are debugging options activated; where does the debug output go?

The test can provoke access violations and other non-recoverable errors and exceptions inside the driver. To successfully debug those failures, run WIA Service under the debugger and run WiaTester. For instructions on installing and using debugger tools, see install.htm on the LDK CD.

Test Log Diagnosis

The test log lists the DDI calls and the variables that are passed in. Any failure of these calls is logged with all pertinent information to help diagnose the problem.

If failure is hard to diagnose, search Windows Hardware and Driver Central at: http://www.microsoft.com/whdc/ for appropriate errata.

Debug Instructions

To help debug the test, sometimes it is more efficient to run only the test case that failed. To do this, read the resulting log file and identify which test script failed (the failed test script is listed by its file name and the extension .wia two or three lines above the actual failure).

Then, run WiaTestGui.exe from the install folder. Under Select a test suite or test script, click Browse to open the Open File dialog box. All test cases are located in the Scripts folder. The name of the test case will give you a hint as to which sub-folder it is located in. For instance, if you want to run TestMethodsdrvDeviceCommand.wia, navigate to Scripts and then Methods. Select the TestMethodsdrvDeviceCommand.wia test case, and then click Open. To run the test case, click the Start Test button.

Command-Line Arguments

The following command-line arguments, except various switches, are part of the test:

WiaTester Usage:

---------------

 

WIATester.exe [-D deviceID] -r {testfile | suitefile} [ {testfile2 |suitefile2} ] [ {-q | -v | -debug} ] [-w] [-i iterations_no] [-p] [-F folder]

 

WIATester.exe -c testfile|suitefile [ {-q | -v | -debug} ] [-w]

 

WIATester.exe -enum [ {-q | -v | -debug} ] [-w]

 

WIATester.exe -fail[n] [ {-q | -v | -debug} ] [-w] [-i iterations_no] [-p] [-F folder]

 

WIATester.exe -last [ {-q | -v | -debug} ] [-w]

 

WIATester.exe -D deviceID

 

WIATester.exe -F folder

 

where:

-r switch means running a test case or a test suite. A test case is a text file with the .wia extension. A test suite is a text file with the .wis extension. A test suite contains a (relative) file path on every row, which represents a test case. The switch is followed by one or two test case files or test suite files. When two files are specified, there are two threads running the tests simultaneously.

-D optional switch specifies the WIA device under test. It should be followed by a WIA device ID.

 

To enumerate the WIA devices available on your system, use:

     WIATester.exe -enum [-w] [-q | -v | -debug]

 

To run a test, specify the device at least once. A configuration file (WIATest.cfg) will be written and the device ID will be saved.

The -q, -v and -debug optional switches set the logging level (quiet, verbose or debug). By default, the logging level is quiet.

The -i optional switch represents the number of times each test will be executed and is followed by a positive integer.

The -p optional switch means the user will be prompted between test iterations

The -w optional switch, when specified, means a log window will be shown. The test execution log will be available at run time in this window.

The -R optional switch specifies a folder in which to save the acquired images and is followed by a folder (relative) path.

The -fail switch re-executes the last n failed tests; n is optional and should be a positive integer.

The -last switch shows a last executed test or suite result.

The -c switch checks a test or a suite for correct syntax.

 

 

The following are WiaTester.exe command-line examples:

WIATester.exe -enum -verbose

Description: enumerates the available WIA devices; the log level is in verbose mode.

 

WIATester.exe -F Acquired_Images

Description: sets the folder to Acquired_Images where the acquired images will be saved.

 

WIATester.exe -D "Image\0001" -r Scripts\Test1.wia -w

Description: the device under test is "Image\0001", executes the "Scripts\Test1.wia" test case, and the log window is enabled.

 

WIATester.exe -D "Image\0000" -r Scripts\Test1.wia Scripts\Test2.wia -w -debug

Description: the device under test is "Image\0000", executes the "Scripts\Test1.wia" test case and the "Scripts\Test2.wia" test case in two separate threads; the log window is enabled and the log level is debug.

 

WIATester.exe -r Suite.wis -i 4 -p -w -quiet

Description: executes the Suite.wis suite file four times every test, the user is prompted between two test executions, the log window is enabled, and the log level is quiet.

 

WIATester.exe -r Suite.wis -p -quiet

Description: executes the Suite.wis suite file, the user is prompted after every test, and the log level is quiet.

 

WIATester.exe -fail 10 -w

Description: executes the last 10 failed or aborted tests (if any), the log window is enabled.

 

WIATester -last

Description: shows last execution summary.

Troubleshooting

1) When using the absolute or relative path for a file or folder and the path contains blanks, enclose the path between double quotes (for example, "C:\Program Files\folder\file.wia")

2) The log information will appear in the console window and in a file named WIATest.wtl.

When the -w switch is specfied, the log is available at runtime.

3) The log application will not close automatically. It can be reused after running a test (for example, if a new WIATester.exe console application is launched and the log window from the previous execution session wasn't closed by the user, then the WIATester application will use it instead of creating a new one).

5) You can navigate through the test failures by clicking the Arrow buttons on the tool bar (click the Next button (->) to go to the next failure, or click the Back button (<-) to go to the last failure.

6) It is recommended to test robustness and possible resource leaks of a WIA driver after it passes the WIA compliance test. Select Stress my WIA driver in WiaTestGui.exe before starting to test your driver.
