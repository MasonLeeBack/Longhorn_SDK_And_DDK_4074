Friendly Name

Photo Print Test

Description

The Photo Print Test is designed to test some simple photo printing abilities of the driver as reported by the driver itself. The test prints out a series of 5 pictures in both landscape and portrait orientation. After the Photo Printing feature is checked into Longhorn, the test will be changed to incorporate it.

Theory of Operation

This test runs as a console application. When it starts, it parses the command line for a specified printer name and, if none is given, prints to the default printer. It queries the printer for the printable area and then takes 5 images and stretches them to fit in both portrait and landscape mode. The output is these 10 pages plus an additional header page that explains to the user what to expect. The purpose of the test is to test basic photo printing. When the new photo printing feature is added to Longhorn, the test will change to test for that.

Environment Requirements

Required Hardware

Standard computer (either X86, AMD64, or IA64) and an installed printer

Required Software

The test requires only a standard operating system.

Tester Knowledge Requirements

Testers must know the correct output of the printer driver that is being tested. For example, the output of a modern color printer is radically different from the output of an older dot matrix printer. Testers must understand the abilities of the printer and driver.

 

Content Quality

Unit Test

The expected output of the test is a total of 10 pages of pictures (5 in landscape mode and the same 5 in portrait mode) and a log file header that explains what should be printed out.  Any failure of these pages to print properly (conforming to the drivers abilities) is a failure of the test.

 

Diagnosability

Setup Verification

Because the test is a single executable file that is copied to your computer, no setup verification is required.

Debugging Activation and Output

This test catches and logs exceptions to the log file (photoprinttest.log). To catch both test tool bugs and driver bugs, you should enable both NTSD and Application Verifier on the test when it is running.

Test Log Diagnosis

The test log is very easy to read and understand. It lists the device driver interface (DDI) calls and the variables that were passed in. Any failure of these calls is logged with all pertinent information. Driver developers should be able to use this information to diagnose the problem.

Test Duration

Depending on the speed of the machine and driver, the test itself should run no more then 5 minutes. However, the printer could take up to 30 minutes to produce the output.

Restart

This test does not require a restart.

Default Configuration

This test tests the default printer.

Interference

You do not need to make any provisions to prevent interference when running this test.

Command-Line Arguments

/p= printer name

Note If the printer name has spaces, you must wrap it in quotes. For example, /p=my printer is correct, but /p=my printer is incorrect.

Troubleshooting

The most common errors would be not having a printer installed or not having a WTTLOG.dll on the system.
