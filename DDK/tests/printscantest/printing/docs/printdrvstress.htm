Description

This tool exercises various printing code paths, focusing on the print driver DDI hooks implementation. In addition, the tool creates a hostile and randomized environment, which is generally known as ‘stress.’ The tool itself does not use fault injection internally.  

Theory of Operation

The tool is a command-line executable. It parses the command line for inputs like printer name, how much time to run (minimum 30 minutes), and randomness percentage. The printer name can be given either as “/printer=My Printer” or simply as a second command line parameter, or omitted. In the latter case, the default printer will be used.

After the tool is launched, it creates a log file and several threads of execution, each of which operate on similar print test objects. After the specified amount of time, the tool stops producing print jobs and writes the result (PASS or FAIL) in the log file. The log file is named “printdrvstress.xml.”

Important note:

This is a robustness test. As such, it is not unexpected that a faulty print driver will crash. A debugger break is also considered “a crash,” though there is no programmatic way to generically detect that condition. The correctness of the log is defined as follows:

-     Case 1: The tool completes successfully AND spooler (spoolsv.exe) is not crashed. In this case, the driver is considered robust.

-          Case 2: The driver crashes in the printdrvstress process. In this case, there will be no log, the log will be empty, or there will be no PASS in the log.

-          Case 3: The driver crashed in the spooler process. This is the worst case, as there could be a “PASS” in the log.

Test Log Diagnosis

If there is no log, or the log does not explicitly state “PASSED,” then the test failed. If a debugger was available, the possible debugger break could explain the specific error.

If there is a log and it states PASSED, then the result is correct only if the spooler service (spoolsv.exe) has not crashed.

Logo Test Specific Information:

Logo Testing Verification

Several orthogonal ways:

1)       By inspecting the log and pass/fail semantics.

2)       By having a debugger attached to the tool and spooler service. Debugger breaks indicate failure.

Test Duration

Minimum 30 minutes. On average, 24 hours would suffice.

Reboot

None required.

Default Configuration

It will test the default printer, with 5% randomness, on the primitives for 24 hours.

Interference

Don’t delete or rename the printer that this tool deals with.

Don’t run simultaneous print jobs on the printer.

Don’t restart the spooler service.

Don’t change spooler settings.

Command Line Arguments

There are several forms of interpreted command lines:

Printdrvstress.exe “My Printer” [/randPercentage=some_positive_integer] [/runTime=some_integer_minutes]

Printdrvstress.exe “/printer=My Printer” [/randPercentage=some_positive_integer] [/runTime=some_integer_minutes]

Printdrvstress.exe [/randPercentage=some_positive_integer] [/runTime=some_integer_minutes]

randPercentage is a decimal positive integer that mandates how random the print tasks should be.

runTime is a decimal positive integer that mandates how much time in minutes the test should run.

The first and the second form of the command line will print to printer with a name “My Printer.” The last form will print to the default printer.

Troubleshooting

Most common problems would result from there being no printer installed or there is no WTTLOG.DLL in the path.
